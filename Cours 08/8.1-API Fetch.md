# L'API Fetch — Fiche théorique

## 1. Qu'est-ce que l'API Fetch ?

L'API Fetch est une interface native du navigateur (et de Node.js ≥ 18) permettant d'effectuer des **requêtes HTTP asynchrones** en JavaScript. Elle remplace l'ancien `XMLHttpRequest` avec une syntaxe plus lisible, basée sur les `Promises`.

> **À retenir** : `fetch()` retourne toujours une `Promise` — il faut donc la gérer avec `.then()/.catch()` ou `async/await`.

## 2. Syntaxe de base

```js
fetch(url, options)
  .then(response => response.json())
  .then(data => console.log(data))
  .catch(error => console.error(error));
```

| Paramètre | Type | Description |
| --- | --- | --- |
| `url` | `string` | L'URL de la ressource à atteindre |
| `options` | `object` | Configuration optionnelle (méthode, headers…) |

## 3. L'objet `Response`

`fetch()` résout sa `Promise` avec un objet `Response`, même en cas d'erreur HTTP (404, 500…). Il faut donc **toujours vérifier la propriété `response.ok`**.

```js
fetch('/api/data')
  .then(response => {
    if (!response.ok) {
      throw new Error(`Erreur HTTP : ${response.status}`);
    }
    return response.json();
  })
  .then(data => console.log(data));
```

### Propriétés et méthodes clés de `Response`

| Propriété / Méthode | Description |
| --- | --- |
| `response.ok` | `true` si le statut HTTP est entre 200 et 299 |
| `response.status` | Code HTTP numérique (200, 404, 500…) |
| `response.statusText` | Message associé au statut (`"OK"`, `"Not Found"`…) |
| `response.headers` | Objet `Headers` avec les en-têtes de la réponse |
| `response.json()` | Parse le corps en JSON → retourne une Promise |
| `response.text()` | Retourne le corps brut sous forme de chaîne |
| `response.blob()` | Retourne le corps comme donnée binaire (images, etc.) |

> ⚠️ Les méthodes `.json()`, `.text()`, `.blob()` ne peuvent être appelées **qu'une seule fois** : le flux est consommé.

## 4. Les méthodes HTTP

### `GET` (lecture)

```js
// Avec async/await — syntaxe recommandée
async function getUser(id) {
  const response = await fetch(`/api/users/${id}`);
  if (!response.ok) throw new Error('Utilisateur introuvable');
  return await response.json();
}
```

### `POST` (création)

```js
async function createUser(userData) {
  const response = await fetch('/api/users', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json'
    },
    body: JSON.stringify(userData)
  });
  return await response.json();
}
```

### `PUT` / `PATCH` (modification)

```js
// PUT remplace la ressource entière
// PATCH modifie partiellement la ressource
const response = await fetch(`/api/users/${id}`, {
  method: 'PATCH',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ email: 'nouveau@example.com' })
});
```

### `DELETE` (suppression)

```js
const response = await fetch(`/api/users/${id}`, {
  method: 'DELETE'
});
// Les réponses DELETE retournent souvent un statut 204 (No Content)
```

## 5. L'objet `options` en détails

```js
fetch(url, {
  method: 'POST',          // GET | POST | PUT | PATCH | DELETE
  headers: {               // En-têtes HTTP
    'Content-Type': 'application/json',
    'Authorization': 'Bearer <token>'
  },
  body: JSON.stringify({}),// Corps de la requête (POST, PUT, PATCH)
  mode: 'cors',            // cors | no-cors | same-origin
  credentials: 'include',  // omit | same-origin | include (pour les cookies)
  cache: 'no-cache',       // default | no-store | reload | no-cache | force-cache
  signal: controller.signal // Pour l'annulation (AbortController)
});
```

## 6. Envoyer différents types de données

### `JSON` (le plus courant)

```js
fetch('/api/data', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ nom: 'Alice', age: 30 })
});
```

### `FormData` (formulaires, fichiers)

```js
const formData = new FormData();
formData.append('nom', 'Alice');
formData.append('avatar', fileInput.files[0]);

fetch('/api/upload', {
  method: 'POST',
  body: formData
  // ⚠️ Ne pas définir Content-Type manuellement — le navigateur le gère
});
```

### `URLSearchParams` (paramètres de requête encodés)

```js
const params = new URLSearchParams({ page: 1, limite: 10 });
fetch(`/api/articles?${params}`); // → /api/articles?page=1&limite=10
```

## 7. Annuler une requête avec `AbortController`

```js
const controller = new AbortController();

// Annuler automatiquement après 5 secondes
const timeout = setTimeout(() => controller.abort(), 5000);

try {
  const response = await fetch('/api/data', {
    signal: controller.signal
  });
  clearTimeout(timeout);
  const data = await response.json();
} catch (error) {
  if (error.name === 'AbortError') {
    console.log('Requête annulée');
  }
}
```

## 8. Gestion des erreurs — ce qu'il faut savoir

La méthode `fetch()` ne rejette sa `Promise` que dans deux cas :

- **Problème réseau** (pas de connexion, DNS introuvable...)
- **Requête annulée** via `AbortController`

Les erreurs HTTP (404, 500…) **ne rejettent pas** la `Promise`. D'où l'importance de vérifier `response.ok`.

```js
// Patron robuste recommandé
async function fetchData(url) {
  try {
    const response = await fetch(url);

    if (!response.ok) {
      // Erreur HTTP → on la gère explicitement
      throw new Error(`Erreur ${response.status} : ${response.statusText}`);
    }

    return await response.json();

  } catch (error) {
    if (error.name === 'AbortError') {
      console.warn('Requête annulée par l\'utilisateur');
    } else {
      console.error('Erreur réseau ou HTTP :', error.message);
    }
    throw error; // On propage pour que l'appelant puisse réagir
  }
}
```

## 9. `CORS` — Comprendre les restrictions *cross-origin*

Le navigateur bloque par défaut les requêtes vers un **domaine différent** (*Cross-Origin Resource Sharing*). Le serveur doit autoriser l'accès via ses en-têtes.

```js
Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
```

En développement, on utilise souvent un **proxy** (Vite, Webpack DevServer) pour contourner CORS.

## 10. Bonnes pratiques

| ✅ À faire | ❌ À éviter |
| --- | --- |
| Toujours vérifier `response.ok` | Supposer que toute réponse est un succès |
| Utiliser `async/await` pour la lisibilité | Chaîner `.then()` en profondeur (callback hell) |
| Centraliser `fetch` dans des fonctions utilitaires | Dupliquer la logique dans chaque composant |
| Gérer l'annulation avec `AbortController` | Laisser des requêtes orphelines en mémoire |
| Afficher un état de chargement dans l'UI | Laisser l'interface sans feedback pendant la requête |
| Typer les données attendues (TypeScript) | Faire confiance aveuglément à la forme des données |

## 11. Comparaison rapide : Fetch vs Axios

| Critère | `fetch` (natif) | `axios` (bibliothèque) |
| --- | --- | --- |
| Installation | Aucune | `npm install axios` |
| Vérification erreurs HTTP | Manuelle (`response.ok`) | Automatique (rejette la Promise) |
| Transformation JSON | Manuelle (`.json()`) | Automatique |
| Annulation | `AbortController` | `AbortController` ou `CancelToken` |
| Intercepteurs | Non natif | Oui, intégré |
| Progression upload | Non | Oui |
| Support navigateurs | Moderne (IE exclu) | Tous avec polyfill |

> **Conseil** : Pour des projets simples ou si vous voulez éviter les dépendances, `fetch` suffit largement. `axios` brille pour les projets complexes avec intercepteurs, gestion centralisée des erreurs, et upload avec progression.

## Récapitulatif visuel du cycle d'une requête Fetch

```text
fetch(url, options)
       │
       ▼
  Promise<Response>
       │
   ┌───┴────────────────────────┐
   │                            │
response.ok = true      response.ok = false
   │                            │
response.json()          throw new Error()
   │                            │
   ▼                            ▼
 data ✅                   catch(error) ❌
```
