# Patron de conception MVC (Model – View – Controller)

## Définition

Le **MVC** est un patron d'architecture logicielle qui sépare une application en trois composants distincts, chacun ayant une responsabilité bien définie. L'objectif est de découpler la logique métier, l'interface utilisateur et la gestion des interactions.

## Les trois composants

### 🗄️ Model (Modèle)

- Représente les **données** et la **logique métier** de l'application.
- Gère l'accès à la base de données, les règles de validation et les calculs.
- **Ne connaît ni la Vue ni le Contrôleur.**
- Notifie la Vue en cas de changement (via des événements ou observateurs).

### 🖼️ View (Vue)

- Représente l'**interface utilisateur** (ce que l'utilisateur voit).
- Affiche les données fournies par le Modèle.
- **Ne contient aucune logique métier.**
- Transmet les actions de l'utilisateur au Contrôleur.

### 🎮 Controller (Contrôleur)

- Fait office d'**intermédiaire** entre le Modèle et la Vue.
- Reçoit les entrées de l'utilisateur, les interprète et décide quoi faire.
- Met à jour le Modèle et/ou sélectionne la Vue appropriée.

## Flux de données

```text
Utilisateur
    │
    ▼ (action)
Contrôleur ──────► Modèle
    │                  │
    │◄─────────────────┘ (données mises à jour)
    │
    ▼
  Vue ──────► Utilisateur (affichage)
```

## Avantages

| ✅ Avantage | Description |
| --- | --- |
| **Séparation des responsabilités** | Chaque composant a un rôle clair et isolé |
| **Maintenabilité** | Modifications faciles sans tout casser |
| **Testabilité** | Le Modèle et le Contrôleur sont testables indépendamment |
| **Réutilisabilité** | Un même Modèle peut servir plusieurs Vues |
| **Travail en équipe** | Front-end et back-end peuvent travailler en parallèle |

## Inconvénients

- Peut paraître **complexe** pour de petites applications.
- Multiplication des fichiers et des couches d'abstraction.
- Le Contrôleur peut devenir **trop volumineux** (*Fat Controller*) si mal structuré.

## Exemples d'utilisation

- **Web back-end :** Ruby on Rails, Laravel (PHP), Django (Python), ASP.NET
- **Web front-end :** AngularJS (variante MVVM)
- **Mobile :** iOS (UIKit suit une approche MVC)
- **Desktop :** Applications JavaFX, Swing

## Variantes du MVC

| Patron | Différence principale |
| --- | --- |
| **MVP** (Model-View-Presenter) | Le Presenter remplace le Controller et gère toute la logique de présentation |
| **MVVM** (Model-View-ViewModel) | Le ViewModel expose des données observables, populaire avec React/Vue.js |
| **MVCS** | Ajout d'une couche *Service* pour la logique métier complexe |

## En résumé

> Le MVC, c'est le principe de **"diviser pour mieux régner"** appliqué à l'architecture logicielle : chaque partie fait une seule chose, et la fait bien.
